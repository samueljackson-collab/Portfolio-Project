# Database Migration Runbook

**Author:** Platform Engineering Team  
**Date:** 2026-01-15  
**Document Type:** Runbook

## Overview

This runbook documents the procedure for migrating the primary PostgreSQL database from version 14 to version 16 on the production cluster. The migration uses pg_upgrade with a blue-green approach to minimise downtime to under 5 minutes.

## Prerequisites

- Access to production AWS console (AdministratorAccess or DatabaseAdmin role)
- pg_upgrade 16 installed on the migration host
- Verified backup from within the last 6 hours (check S3 bucket prod-db-backups)
- Change management ticket approved (CAB approval reference in ticket description)
- Maintenance window confirmed with stakeholders — Saturday 02:00–06:00 UTC
- Runbook reviewed by DBA lead and signed off
- Rollback plan tested in staging environment

## Procedure

1. **Enable maintenance mode**
   Set MAINTENANCE_MODE=true in SSM Parameter Store and wait for ALB health checks to drain existing connections (approx. 30 s).
2. **Create final pre-migration snapshot**
   aws rds create-db-snapshot --db-instance-identifier prod-postgres-14 --db-snapshot-identifier pre-migration-$(date +%Y%m%d%H%M)
3. **Stop application write traffic**
   Scale the app-server ECS service to 0 tasks: aws ecs update-service --cluster prod --service app-server --desired-count 0
4. **Verify no active connections**
   psql -h prod-db.internal -U postgres -c "SELECT count(*) FROM pg_stat_activity WHERE state='active';" — expected output: 0
5. **Run pg_upgrade dry-run**
   /usr/lib/postgresql/16/bin/pg_upgrade -b /usr/lib/postgresql/14/bin -B /usr/lib/postgresql/16/bin -d /var/lib/postgresql/14/main -D /var/lib/postgresql/16/main --check
6. **Execute pg_upgrade**
   /usr/lib/postgresql/16/bin/pg_upgrade -b /usr/lib/postgresql/14/bin -B /usr/lib/postgresql/16/bin -d /var/lib/postgresql/14/main -D /var/lib/postgresql/16/main
7. **Update RDS parameter group**
   Attach the rds-pg16-production parameter group to the instance and reboot: aws rds modify-db-instance --db-instance-identifier prod-postgres-16 --db-parameter-group-name rds-pg16-production --apply-immediately
8. **Run post-upgrade ANALYZE**
   /usr/lib/postgresql/16/bin/vacuumdb --all --analyze-in-stages -h localhost -U postgres
9. **Update connection string in SSM**
   aws ssm put-parameter --name /prod/db/connection_string --value 'postgresql://app:***@prod-postgres-16.internal:5432/appdb' --overwrite
10. **Scale application back up**
   aws ecs update-service --cluster prod --service app-server --desired-count 4
11. **Disable maintenance mode**
   Set MAINTENANCE_MODE=false in SSM Parameter Store.
12. **Monitor error rates for 15 minutes**
   Check CloudWatch dashboard 'prod-app-errors'. Alert threshold is > 1% error rate over 5 minutes.

## Validation Checks

1. Application health endpoint returns HTTP 200: curl -s https://api.example.com/health | jq .status
2. Database version confirmed: psql -c 'SELECT version();' — must include 'PostgreSQL 16'
3. Row counts match pre-migration snapshot for tables: users, orders, products, sessions
4. Write operations succeed: run smoke test suite (make smoke-test)

## Rollback Procedure

1. Scale down application services to 0 ECS tasks
2. Restore pre-migration snapshot to prod-postgres-14 instance
3. Revert SSM connection string to prod-postgres-14.internal endpoint
4. Scale application services back to 4 tasks
5. Verify application health and disable maintenance mode
6. Notify stakeholders and create follow-up incident ticket
